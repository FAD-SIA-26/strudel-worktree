Spec Walkthrough

What ORC Is

ORC is a single-process orchestration system that turns one user goal into a visible tree of parallel git-worktree-based Codex runs, then reviews, compares, previews, and merges the winning results.

The shortest version: one Mastermind plans the run, one Lead owns each section, several Codex workers try variants in parallel, a Reviewer picks a winner, and a serialized Merge Coordinator integrates winners into a run branch.

What is durable?

SQLite. The event_log records what happened, and projections record current state.

What is ephemeral?

The in-memory command queues and event bus. They exist only while the Node process is running.

What is the real code artifact?

Git worktrees and branches. Each worker owns a separate branch and isolated checkout.

What does the human do?

Watch, compare, preview, approve, drop, and make high-level product decisions when automation gets stuck.

End-to-End Run Flow

  1. The user starts a run with something like orc run "build a lo-fi track". ORC creates a run identity and a dedicated integration branch such as run/<id>.
  2. The Mastermind decomposes the goal into sections. If a workflow template matches, ORC uses that structure first; otherwise the Mastermind decomposes freely.
  3. For each section, ORC creates a Lead. Each lead creates section-local worker prompts and spawns several parallel worker variants.
  4. Each worker gets its own branch, its own worktree, its own plan file, and its own Codex process.
  5. Workers emit progress and state changes. The dashboard updates live over WebSocket from the in-memory event bus.
  6. The Reviewer compares successful worker outputs and picks one winner for the section.
  7. The Merge Coordinator serializes merges into the run branch. If a merge conflicts, ORC spawns a bounded merge-fix worker instead of asking the user to touch code.
  8. When all non-optional sections are complete and their merges are done, ORC fast-forwards the run branch to main and marks the orchestration complete.

Who Does What

Mastermind

Top-level orchestrator. Plans the run, creates leads, monitors overall progress, owns final completion/failure decisions.

Lead

Owns one section. Runs the PM step, spawns workers, collects results, asks reviewer for a winner, requests merge.

PM Agent

LLM planning step that turns a section goal into distinct worker prompts. It is not a long-lived process.

Worker

The only real spawned Codex subprocess in the MVP. Works inside one isolated git worktree.

Reviewer

LLM arbitration layer that chooses the best successful worker output for a section.

Merge-Fix Worker

A bounded repair worker spawned only for code-level merge conflicts. It is not allowed to re-plan the product.

Important distinction: in the MVP, Mastermind and Lead are orchestration roles inside the Node process. The main external spawned agents are Codex workers.

How State Actually Works

1. SQLite is the truth

The system writes every meaningful state transition into event_log, then updates projection tables such as tasks, runs, worktrees, merge_queue, previews, and artifacts.

2. Command transport and event fanout are separated

Command Queues

Transport only

In-memory per-role async queues. Used for downward commands such as Mastermind → Lead and Lead → Worker.

Event Bus

Observability only

Publish-only in-memory event emitter. Used to fan out state changes to the dashboard after they are written to SQLite.

SQLite transaction:
  append event
  update projection

then:
  publish same event to in-memory event bus
  WebSocket broadcasts to browser clients

3. There are no direct worker-to-worker calls

All cross-role interactions are visible as journaled events like CommandIssued, WorkerProgress, ReviewComplete, MergeRequested, MergeConflict, and MergeResolved.

What Happens Inside A Worker

  1. A lead chooses a branch name like feat/rhythm-v1 and creates a corresponding worktree under .worktrees/.
  2. The worker gets a context bundle: task prompt, strategy, retry history, path to run plan, path to lead plan, path to worker plan.
  3. CodexCLIAdapter spawns a real codex subprocess in that worktree.
  4. Output is appended to .orc/.orc-session.jsonl for retry/resume context.
  5. On exit, the adapter writes .orc/.orc-done.json as the terminal marker.
Internal Codex subagents: the spec allows a worker to use Codex subagents internally, but the orchestrator treats that as one opaque worker run. Only top-level workers are scheduled and visible.

How ORC Detects Trouble

The watchdog runs every 5 seconds and derives worker health from three signals:

Stdout inactivity

If no new progress appears for longer than the threshold, the worker becomes stalled.

PID liveness

If the process is dead and no done marker exists, the worker becomes zombie.

Completion marker

If .orc/.orc-done.json exists, the worker becomes terminal: done or failed.

Health states used in the UI:

queued running stalled zombie done failed

How Merging Works

ORC does not merge worker winners directly into main.

  1. The run starts its own integration branch: run/<orchestration-id>.
  2. Each lead emits a MergeRequest for its winning worker.
  3. The Merge Coordinator serializes merges FIFO into the run branch.
  4. If a merge is clean, the queue continues.
  5. If a merge conflicts, ORC spawns a merge-fix worker on a repair branch like repair/<merge-id>-v1.
  6. If automated repair keeps failing, the human only makes a high-level choice: retry another candidate, skip the section, or fail the orchestration.
Important: the user never resolves code-level merge conflicts. ORC keeps the human at the product-decision layer, not the git-conflict layer.

How Resume / Recovery Works

orc resume is MVP best-effort continuation from SQLite plus worktree reconciliation.

  • Load current projections from SQLite.
  • Inspect each worktree for .orc/.orc-done.json.
  • If the marker says done or failed, trust that terminal state.
  • If there is no marker but the worktree is dirty, treat it as interrupted in-progress work and resume with prior session context.
  • If there is no marker and the worktree is clean, restart from scratch.

What Is Actually MVP

In MVP

SQLite, worktrees, Codex workers, watchdog, dashboard, previews, compare/approve/drop controls, workflow templates, serialized merge coordinator, AI merge-fix worker.

Post-MVP

Redis, outbox processor, distributed coordination, security agent, richer strategic retries, and stronger cross-lead context sharing.
